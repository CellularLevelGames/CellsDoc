3 Networking
4 ==========
5 
6 The Photon Framework
7 --------------------
8 We use the Photon PUN library and service for our multiplayer. Benefits are an abstraction of the lower-level
9 networking stuff and server-based connections, i.e. no Peer-To-Peer and NAT problems.
10 
11 Photon offers different means of network communication and I'll introduce them here:
12 
13 ### PhotonView
14 This components can be added to GOs to enable instance-based syncing as follows:
15 
16 * The object can be spawned and deleted on all clients in sync and Photon maintains the logical connection
17  between each copy on the clients.
18 * Remote-Procedure-Calls (RPCs) can be called for scripts on that object. These allow a function to be
19  called from one client (mostly the owner/creator of the GO) to be activated on all clients.
20 * Synchronization of data from other components. Photon offers pre-made syncing scripts, for example for the
21  transform, which thus syncs movement.
22  Custom components have to implement a certain Interface to receive callbacks to read and write data to sync.
23 
24 ### PhotonEvents
25 Photon also features an event system, where each event has a unique ID and can transport data in the shape
26 of an array of *object*-type objects. To make this easier we have a custom extension for this, called
27 net::NetEvent, which is explained below.
28 
29 ### Client Properties
30 This mechanism saves data under a hashed string as identifier for one client. All clients can access the data
31 of other clients, making this a good tool to sync client-specific data (e.g. a 'Readiness' flag) across all
32 clients. The client data is kept even through disconnects and room changes, I think.
33 For BomberUnits, this was used to save information on the local players for each client, including their
34 chosen color, model, etc...
35 
36 #### Room Properties
37 Client assemble within one room to be able to play a game together. This room can also maintain synced data,
38 akin to the client properties. This could for example be used to share common settings for the game, such as
39 the selected level.

42 What needs to be synced?
43 ------------------------
44 I think there are three important categories of what needs to be synced:
45 * Game Logic: Everything related to the state of the game of course needs to be in sync. This encompasses
46  general things, like readiness indication in matchmaking an common scene transitions, as well as important
47  events, such as the demise of unit.
48 * Fairness-related HMI:
49 * Nice-to-have HMI things: Everything that is not needed according to the above categories, but gives the
50  players more information, such as having health bars for units.
51 
52 The means to do the syncing should be made as easy and clear as possible, as the networking stuff can become
53 quite tangled quickly anyway. It has to be decided on a case-by-case basis how this is to be done.
54 
55 All synced elements and the concepts used are explained on a subpage: \subpage page_synced_elements

57 The NET Management layer
58 ------------------------
59 In addition to the above mechanism, communication with the Photon API and the current network is needed
60 For this the net::NetManager singleton instance manager receives callbacks, which are then raised as events
61 for all relevant users to pick up.
62 
63 The net::NetAdapter static class offers our own interface to the Photon static API functions. It provides,
64 for example, functions to create and destroy objects networked, i.e. in sync on all clients.
65 
66 Finally, the net::NetStateMachine implements the different states the connection can be in as an Animator-
67 based SM. Most states subscribe to events from the net::NetManager to decide how the SM transitions.
68 
69 Custom expansions on the provided networking API
70 ------------------------------------------------
71 
72 The net::NetEvent class is designed to work like a UnityEvent, while offering the capability to invoke them
73 networked on all clients. Each created instance automatically registers itself with a unique ID to the Photon
74 event system. Each event can carry data types which implement the net::INetDataPkg interface.
75 Implementing this interface means casting all data held within the type to *object* and back, so it can be
76 handled uniformly within the NetEvent.
