About the game

Cells is intended to be a new type of tactical command game. We want to combine RTS-like components, such as commanding units and gathering resources with Two-Stick-Shooter elements, where a a single player character fights and interacts with the world. The goal is to finally destroy the home bases or player characters of your enemies.

We want to support only competitive play for now, but aim to also allow for couch-coop etc. using controller input in the future.

The setting is that of microbiology, with units essentially being bacteria and the like. We think this lore allows for many interesting game mechanics while working for our (currently) limited artistic capabilities.

Interested? Read the Getting Started page so set things up. For more details on what there is to do, read on :)

Game Design

The core of any project: figuring out what we actually want. In order to get going, we need at least some basic ideas, preferably with an interesting twist. Then we need to refine these ideas to answer questions such as:

  • How will playing this be fun?
  • What interesting choices does this element give to the player?
  • What are the core elements and themes of the game? Are they clearly visible?
  • What shall the Player's experience be like?
  • What should the graphical style be like? (2D, 3D, Rendering, Poly count...)
  • What main functionality will the SW have to provide? (Think e.g. multiplayer...)
  • What other, crazy ideas do we have? Do they fit in?

The tool to use here is mainly the Game Design project in GitHub, where you can add and sort ideas. You can also start discussions in our GitHub Team (CellsDevTeam) or via What's App. Be sure to cement decisions in .md files for doxygen :)

Planning & Teamwork

So, imagine we have quite a big list of tasks, big and small, crazy ideas and stuff we just want to try to see whether they work. We need to:

  1. Decide what has to be done in which order from a technical perspective.
  2. What we want to do first of the things that require no special order. What are the ratios of gain vs. effort for the features we want to add?
  3. Who wants to do what? Is everyone happy with what he is doing and where we are going? Does anyone need help?

Also, stuff needs to be taken care of:

  • When and how do we meet?
  • How do we want to collaborate? Are there any rules?
  • Most importantly: How can we improve?

One tool to support this is surely the project tab and issue tracking as well as Team area of GitHub, where our code is hosted anyway. It offers a way to collect, sort and prioritize ideas as notes, track tasks, and rant about them in strong language. Information not related to ideas or tasks may be tracked using these nice Markdown pages.

Rendering & Shading

Unity allows for a number of different rendering pipelines and it is a field in its own to find out which suit our needs best. This includes quite some Google usage and testing stuff.

Making Shaders is actually quite easy using the Shader Graph package, where one can design the look of surfaces with a myriad of options. An typical example would be creating a water shader.

Art Design & Modelling

If this is not a text adventure, we will need to show some art, be it 2D or 3D. How can we achieve this without hiring Weta Workshop? How do we achieve a reasonable look with comparably low effort? The current take on this is to have low-res models/art and high-res effects. This is a common thing to do in the industry seemingly when you are low on budget...

The tools to support this are manyfold, with many specialized solutions for the different artsy tasks, depending on what we want. Simple models can be made for example with AssetForge, while arbitrarily complex stuff can be done with Blender. Maps and textures can be found, also free, all over the web. The final look will also largely depend on the Postprocessing Stack, allowing cool looks for low-poly models. Sprite drawing also has a range of tools, like Photoshop or Aceprite.

Rigging & Animation

Rigging is the task of providing a virtual skeleton, or rig, for your models. This allows to deform the models to assume natural looking poses. A sequence of such poses forms an Animation, which is a kind of asset importable and replay-able in Unity. Animations can be used for other things as well though, for example to animate menu items or even to implement state machines, e.g. for enemy behaviour.

Rigging is typically supported by modeling SW, such as Blender. Animations may be done in the Unity editor as well, while the final chaining of animations with animation state machines is a task of game development, also supported graphically by the Unity Editor.

Level Design

This is what the title implies, and you have probably seen good and bad levels abound yourselves. It is a bit of a cross-over task:
The design aspect of this is a part of the Game Design, especially when thinking of cool level features, which will need support from a Scripting perspective.

For the actual modeling of a level Scene, using mainly the Unity Editor as Tool, you will combine Models as well as maybe create your own stuff, as a Modeling guy would do. 2D level design takes advantage of Unity's Tilemap system, where one can have intelligent sets of tiles to# draw a level with.

Scripting

This is the art of learning what the Unity API has to offer, how to use that to design an understandable and reusable SW architecture and - of course - coding the thing in C# (or Java Script alternatively). The scripts bring the game objects to life, let you play with friends, produce tons of log output, and so on...
Game development may sound like it is all about Coding/Scripting, but as you see, there are many daunting tasks each worthy ofa whole profession in the industry.

The tools to do that are Unity and a supported IDE, such as Visual Studio or Visual Studio Code. Have a look at the Tooling section (Tooling) for how to set those up. There is no black magic involved, as Unity pretty much has the whole toolchain, up to deployment, built-in.

Effects & Lighting

This category wraps up some of the typical features found in Game Engines: particle systems, sound effects and the relevant scripting making effects affecting the game world. This is also a bit of a crossover task, and you get to play with explosions. Also in the effects category are screen effects, supported by the rendering pipeline used, which is quite the topic of its own.

The Unity editor may be all the tool you need to get started. Also Unity gives away quite a lot of free-to-use effects we can start with. Building on that, you may need to roam the Net and use some tools of the other disciplines to create you very own uber-effects!

Lighting again is a thing of its own, with quite some renderer support in the case of 2D. There are a lot of tutorials out there on lighting in 2D or 3D, always dependent on the used Render Pipeline.

Post processing

In Post processing, meaning after the basic rendering of the frame, it is possible to apply further transformations to apply effect or improve style and image quality. Typical tasks are Anti-Aliasing or the Bloom effect. This has to be applied, tuned and tested. From the tooling perspective, the basic effects are available directly in Unity. But it may be possible to churn out some extra stuff with some research, who knows.

DevOps

If we decide to have, e.g. build automation, common tool configurations (such as for Doxygen in the doc folder), etc. - someone will have to look after that. Someone willing to be the all-important enabler of the whole team. That someone could be you!

The skills learned/needed here are probably no special to game development, and the tools are manyfold, depending on what the team deems worthy automating. Most important though are the GitHub Actions and workflows, which provide e.g. checks on Pull Requests.

Testing

No time for the other stuff? No worries - we need people that from time to time have a look at the current state and tell us what we f**** up. A pat on the back is also helpful sometimes.

And then, there is proper testing, including Unit Tests. It may not seem reasonable for the dynamic nature of game scripting code, but it is very well possible and supported by Unity's own set of tools. The scripters will kiss the feet of the guy who tells them how to use those tools. At the very latest when complexity increases and devs become afraid of breaking things accidentally.
