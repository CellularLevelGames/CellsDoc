3 Synced Elements
4 ===============
5 
6 Game State
7 ----------
8 The basic game state and behaviour is visible from and steered by the GameStateMachine.
9 In some situations the SM needs to be advanced in sync, for which a net::NetEvent has been added to the
10 management::GameStateMachine. The NetEvents carries the Transition name information and is listened to by
11 the SM itself to trigger the transition on each client.
12 
13 Scene Loading
14 -------------
15 Scene loading is synced automatically by an according setting for Photon. It makes all clients transition
16 to the same scene the master is loading.
17 It has to be ensured though, that all clients have finished loading the scene before further scripts are
18 executed. For this, a "SceneLoaded" property was added to the client properties table, which is kept in sync
19 by photon. The master resets this flag to 'false' for all clients and these set it to 'true' once they have
20 finished their loading. The level::LevelManager monitors these flags and raises an event once they are 'true'
21 for all clients.
22 
23 Player Character spawning
24 -------------------------
25 The point in time and the position of players upon match start needs to be synced. The host decides which
26 spawn position is assigned to each player and needs to transmit that information to each client.
27 That networking step should be kept as far as possible within the responsible management::GameStateMachine
28 state, management::MatchInitState.
29 
30 The current approach is to send a net::NetEvent with the index of the spawn position within the spawnPositions
31 array of the level from master to each client specifically. The state itself subscribes to these events, so it
32 is received on every client.
33 
34 Instantiation of course requires a PhotonView on the player prefab, but that is needed for synchronization
35 anyway.
36 
37 Player Character movement and animation
38 ---------------------------------------
39 Currently, the default PhotonView components are used to sync the position as well as certain parameters of
40 the animator to sync animation state transitions.
41 
42 \todo To reduce network load, we should only sync the rigidbody speed and completely base animation on
43 monitoring the current speed instead of setting parameters according to the current input. This would make
44 animations synced automatically without additional transmissions (apart from rigidbody states).

46 Unit Attacks and Damage
47 -----------------------
48 To enable dodging, the hit registration should be done on the client owning the hit unit. For this:
49 
50 * Attack animation is triggered on all clients via RPC
51 * Upon the instance of damage, the animation calls a callback on each client
52 * Every client checks, whether one/multiple of their units are hit by the attack and applies damage.
